# Phase 2 Plan 01: Content Schema Definition Summary

Defined 8 typed Astro Content Layer collections in `src/content.config.ts` using Zod 4 idioms — 5 list collections (glob loader) and 3 singletons (file loader) — establishing the type contract that downstream Wave 2/3 plans will author entries against.

### Shared Helpers

- `iconSchema = z.string().regex(/^[a-z0-9-]+:[a-z0-9-]+$/, { error: '...' })` — enforces the Iconify "prefix:name" shape per D-16.

Both factored as helpers (not inlined) — CONTEXT.md's "Claude's Discretion" allowed either; the reuse count made factoring clearly cheaper to read and to extend.

- **Found during:** Task 1 verification.
- **Issue:** The plan's `<automated>` block contains `grep -c 'defineCollection' src/content.config.ts | awk '{ if ($1 == 8) ... }'`. - **Fix:** Honored the acceptance-criteria text (the canonical specification) over the off-by-one bash script. The text-level acceptance gate passes; the bash regex would not pass against any faithful copy of the RESEARCH.md recipe.
- **Files modified:** None — this is a plan-authoring bug to flag, not a code change.
- **Commit:** N/A.

## Helpers: Factored vs Inlined (per plan's `<output>` ask)

Both `trimmedString` and `iconSchema` were **factored as module-scope helpers**, not inlined.

- `trimmedString` is invoked 24 times across the 8 schemas (counted via grep). Inlining `z.string().trim().min(1)` would add 24 × ~24 chars = ~580 chars of repetition vs the helper's 1-line definition + 24 × `trimmedString()` call-sites. Clear net win.
- `iconSchema` is referenced 2 times (in `about.social[].icon` and `skills.categories[].items[].icon`). The 4-line regex+message constant is structurally identical at both call sites; sharing the regex literal also avoids the risk of one site drifting from the other when D-16 evolves.

Both decisions align with CONTEXT.md's "Claude's Discretion" allowance on shared helpers.
